# 用户反馈与迭代机制

用户在使用过程中的反馈和需求，推动Skill仓库持续改进。

## 运作机制

```
用户使用中遇到问题/有新需求
           ↓
    识别反馈类型
           ↓
    记录到本文件
           ↓
    分析：Skill问题 / 模板不足 / 新需求
           ↓
    更新对应Skill/模板/trigger
           ↓
    用户验证改进效果
```

## 反馈类型

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| Skill不匹配 | 触发词对不上用户说法 | 更新trigger |
| 模板缺失 | 缺少某种场景的模板 | 补充模板章节 |
| 输出不实用 | AI输出的内容不符合实际 | 修订Example |
| 新场景 | 现有Skill覆盖不了 | 新增Skill |
| 流程问题 | Pipeline/Gate不合理 | 修订流程 |

### 记录位置

所有用户反馈记录在本文件末尾的「反馈汇总」章节。

## 用户个性化扩展

用户可在自己的分支中扩展：

### 1. 自定义Trigger

在Skill的frontmatter trigger中追加用户自己的触发词：

```yaml
trigger:
  - "筛选基金"      # 原有
  - "帮我选基"       # 用户自定义
  - "基金怎么挑"     # 用户自定义
```

### 2. 自定义模板

在Skill的Example章节追加用户自己的案例：

```markdown
## Example

...（原有案例）

---

### 你的案例

**客户：** [你的客户描述]
**场景：** [你的使用场景]
**输出：** [你期望的输出格式]
```

### 3. 自定义Gate

用户可调整Gate阈值（需在Skill中注明）：

| Gate | 原阈值 | 用户可调整 |
|------|--------|-----------|
| Gate 4 集中度 | 单产品<30% | C3客户 |
| Gate 5 换手率 | 年换手<200% | 可按需调整 |

## 贡献指南

### 如何提交反馈

1. 在本文件「反馈汇总」表格中新增一行
2. 填写：日期、场景、问题、建议
3. 设置优先级（high/medium/low）
4. 等待处理或自行处理

### 如何更新Skill

1. 反馈确认为有效需求后
2. 修改对应Skill的SKILL.md
3. 在「迭代记录」中记录更新
4. 如有必要，更新README.md

### 更新检查清单

- [ ] Skill名称是否合理
- [ ] trigger是否覆盖用户说法
- [ ] Output Format是否满足需求
- [ ] Example是否有参考价值
- [ ] README是否需要同步更新
